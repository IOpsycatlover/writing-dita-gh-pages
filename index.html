Writing with DITA Topics:

<task>, <concept>, & <reference>

Topic Types:

  • <task>:
    Tells how to do something, and usually includes numbered steps -- a procedure.

  • <concept>:
    Describes something.

  • <reference>:
    Explains details of interest about things and materials, and usually includes information in tables or lists.

<task>

What is a <task>?

  • Topic dedicated to procedural information
  • <concept> and <reference> topics support task topics, but are written in different .dita files.

<task> best practices

  • One procedure per topic
  • Create subtasks for longer procedures
  • Combine short, sequential steps
  • Every step should use an imperative sentence
  • Every <task> element should have a meaningful ID attribute: <task id="espresso-connect">

Common <task> elements

DITA ELEMENT DESCRIPTION
<title> Provides the topic title
<shortdesc> Introduces the task
<context> Background information for users to understand task
<steps> Contains ordered steps for the entire procedure
<steps-unordered> Contains unordered steps for the entire procedure
<steps>, <step>, and <cmd> Contains the steps for the task
<substeps>, <substep>, <cmd> Contains the substep for the step
<info> Provides information users need to complete the step
<stepresult> What happens when step is completed
<stepxmp> An example of what happens when step is completed
<choices> and <choice> Displays choices in a bulleted list
<choicetable> Displays choices in a 2-column table. Describe steps for each choice
<postreq> Info about what must be done after task is completed
<example> Example that tells or supports the task
<result> Expected outcome when task is done

<concept>

What is a <concept>?

  • Supports the goals of the task topics
  • Introduces tools/technology
  • Digs deeper into particular component/property

<concept> best practices

  • Describe one concept per topic
  • Use noun-based phrases for titles
  • Use <section> elements to break-up longer concepts
  • Use titles for each section
  • Use images or other media to aid your concepts
  • Use the <term> tag sparingly for most prominent cases of special terminology / jargon.

Common <concept> elements

DITA ELEMENT DESCRIPTION
<title> Provides the topic title
<shortdesc> Introduces the topic
<conbody> Contains the body of the topic
<section> Organizes content in a topic into sections
<sl> Displays a list of short or simple items
<ul> Displays a content as an unordered bulleted list
<dl> Displays a list of terms or short concepts and their definitions
<fig> or <img> Provides a figure and caption so that you can insert graphics
<term> Highlights new terms

<reference>

What is a <reference>?

  • Supports goals of the task topics
  • Dig deeper into particular component/property

<reference> best practices

  • Describe one type of reference information per topic
  • Use noun-based phrases for titles
  • Write and design for scannability
  • Be consistent about design/elements across topics
  • Use the correct type of table (table vs. simple table, vs. properties)
  • Difficulty: Modularize reference info or not. Rule of thumb: If more than 2 pages, break it up, logically.
  • All <reference> elements require a <section> element
  • Do not "paper" (p. 60) your reference topics. Avoid writing topics that the user does not need. (Example: UI icons.)

Common <reference> elements

DITA ELEMENT DESCRIPTION
<title> Provides the topic title
<shortdesc> Introduces the topic
<refbody> Contains the body of the topic
<section> Organizes content in a topic into sections
<table> Contains the table content
<simpletable> Contains the table content that doesn't require a title
<properties> Lists properties in a table by type, value, and description
<ul> Displays a content as an unordered bulleted list
<dl> Displays a list of terms or short concepts and their definitions
<example> Contains example content that explains or supports the topics
<parml> Displays parameters in a format similar to a definition list
